## TL;DR

Different LLMs are good at different things, so people had to hand-craft the multi-agent workflow each time. **The Conductor is a 7B LLM trained with RL to dynamically emit a natural-language workflow (instruction · assigned worker · visibility) for every input.** Communication topology, prompt engineering, and recursive calls all emerge naturally from end-to-end reward maximization, and the Conductor reaches an average of 77.27% across seven reasoning benchmarks — beating GPT-5 (74.78%) and Gemini 2.5 Pro (70.97%).

## Problem

Frontier LLMs already approach top human experts on many tasks, but unlocking that latent ability requires manually designed agentic workflows — bespoke prompt, refinement, and verification patterns are core IP for commercial AI products, and academia continues to chase prompting tricks. Worse, **no single LLM is universally best** (Chang et al. 2024): each model specializes in different domains. So someone has to redesign which model goes where, in what order, every time. This paper asks whether that meta-design itself can be learned by an LLM via RL — and whether the resulting auto-conductor can beat the multi-agent scaffolds humans build.

## Core Idea

While prior multi-agent work has partially learned routing (which model to send to) or aggregation (how to merge answers), **the Conductor is a single 7B LLM that designs the entire workflow in natural language.** Key differentiators:

- **Workflow as natural-language output**: For each step the Conductor emits (1) a natural-language instruction, (2) the worker agent that should receive it, and (3) which previous responses that worker can see. Up to 5 steps per workflow, with communication topology decided per input.
- **End-to-end reward maximization**: prompt engineering, refinement, verification, and meta-prompt optimization emerge from RL training without dedicated modules. Starts from Qwen2.5-7B; converges in 200 GRPO iterations with batch 256 and no KL regularization.
- **Trained over randomized worker pools**: Worker LLM sets are randomized at training time so the Conductor adapts to arbitrary open/closed-source agent combinations.
- **Recursive self-orchestration**: Allowing the Conductor to select itself as a worker enables dynamic test-time scaling.

## How It Works

<Collapsible title="How It Works">

### Workflow definition

For an input problem $q$, the Conductor outputs a workflow $W = \{s_1, s_2, \ldots, s_T\}$ ($T \le 5$) in natural language. Each step $s_t$ has three components:

- **Natural-language instruction** $i_t$: the subtask the worker should perform
- **Assigned worker** $w_t \in \mathcal{W}$: chosen from the worker pool $\mathcal{W}$
- **Visibility** $V_t \subseteq \{s_1, \ldots, s_{t-1}\}$: which prior step responses this worker can see

This formulation captures communication topology naturally — seeing all prior responses gives sequential refinement, seeing some gives parallel + selective communication, seeing only itself gives isolated execution.

### Training algorithm

GRPO (Group Relative Policy Optimization) on Qwen2.5-7B base. AdamW, batch size 256, 200 iterations, no KL regularization. Worker pools are randomly sampled per training input so the Conductor learns to adapt to arbitrary pools.

### Training data

Just 960 problems — MATH (competition), MMLU (multitask), RLPR (real-world reasoning), LiveCodeBench V1 (code generation). Four domains kept deliberately small to test the hypothesis that a strong worker pool sidesteps the canonical exploration problem of small-model RL.

### Worker pool

- **Closed**: Gemini 2.5 Pro, Claude Sonnet 4, GPT-5
- **Open**: DeepSeek-R1-Distill-Qwen-32B, Gemma3-27B-instruct, Qwen3-32B

### Recursive extension

The base Conductor is further finetuned to allow selecting itself as a worker. At test time, the Conductor sees first-round results and triggers a second Conductor call — a new form of dynamic test-time scaling.

</Collapsible>

## Key Results

State of the art on seven reasoning benchmarks, average 77.27%. Headline numbers:

- **GPQA Diamond**: 87.5 (Gemini 2.5 Pro 84.8, GPT-5 82.3)
- **LiveCodeBench**: 83.93 (GPT-5 82.90)
- **MATH500**: 99.4 (GPT-5 99.0)
- **MMLU**: 94.1 (GPT-5 93.5)
- **AIME25**: 93.3 (GPT-5 90.8)
- **BigCodeBench**: 37.86 (Gemini 2.5 Pro 37.51, GPT-5 32.75)
- **RLPR**: 44.75 (GPT-5 42.20)

In in-distribution evaluation the Conductor surpasses Mixture-of-Agents, MasRouter, and 5-turn reflection baselines by substantial margins, and **matches their performance with far fewer agent calls** (Figure 5).

Additional findings:
- **Randomized-pool training yields OOD generalization** (Figure 6): strong performance on open-model subsets the Conductor barely saw during training.
- **Adapts agent count to task difficulty** (Figure 8): 2 agents on simple tasks like MMLU, 3–4 agents on LiveCodeBench.
- **Recursive calls give additional gains** (Table 2): self-orchestration becomes a new test-time scaling axis.
- **Worker distribution reflects quality** (Figure 10): recursive rounds redistribute selection toward Claude and Gemini — the Conductor implicitly learns a worker-quality ranking.

## What It Achieves and Where It Falls Short

**New achievements**:
- A 7B model, trained with RL, surpasses GPT-5 and Gemini 2.5 Pro by *orchestrating* them — among the early demonstrations that multi-agent collaboration can be unlocked by RL.
- Communication topology, prompt engineering, and verification strategies emerge end-to-end from reward maximization without dedicated modules.
- Recursive self-orchestration introduces a new test-time scaling axis.

**Limitations**:
- *Author-stated*: Reliance on expensive frontier LLMs may exacerbate the economic divide in AI access.
- *Author-stated*: Up to 5-step workflows — longer-horizon tasks require further extension.
- 🤖 *AI analysis (generality)*: All seven benchmarks are knowledge / short-form reasoning. Generalization to truly *agentic* settings — open-ended coding (IDE integration, multi-step debugging), long-form writing / research, external tool use — remains untested. Whether SOTA on short-form reasoning transfers to long-horizon agent work needs separate evaluation.
