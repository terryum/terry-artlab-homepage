`#-5` · 스킬, 에이전트, 그리고 멀티에이전트 슬랙

## 탐색 주제

- 에이전트는 무엇인가 — 정의의 구성 요소
- 에이전트와 Skill은 어떻게 다른가 — 어디까지가 Skill이고 어디부터 Agent인가
- Claude Code에서 쓰는 에이전트와 슬랙에서 쓰는 에이전트가 같은 것인가
- 슬랙을 활용한 멀티에이전트의 실제 쓰임새와 구현 테크닉
- Claude API 단독으로 어디까지 가능하며 비용 범위는 어느 정도인가

## 레슨 요약

- 에이전트는 역할·기억·도구 권한·트리거를 가진 "작업자"다. Skill은 필요할 때 로드되는 "절차서". 한 Agent가 여러 Skill을 품고 시작할 수 있으며, 단순 절차 반복은 Skill과 스케줄러만으로도 충분하다.
- 에이전트 정의는 실행 환경과 무관한 공용 자산 한 벌로 둔다. Claude Code와 슬랙은 같은 정의를 로드하되, 환경마다 얇은 어댑터만 새로 만든다 — 정의와 실행기의 분리가 이 구조의 핵심.
- 슬랙은 에이전트가 생각하는 곳이 아니라 대화 버스다. 추론·장기 메모리·도구 호출은 바깥의 제어 평면(Obsidian + Claude Code + MCP 서버)에서 일어나고, 슬랙은 `@mention`·DM 이벤트 전달과 결과 렌더만 담당한다.
- 슬랙 멀티에이전트의 실제 쓰임새는 "AI끼리의 토론"이 아니라 "채널에서 전문 에이전트를 호출해 팀이 답을 검토"하는 협업 패턴이다. 2026년 Dust의 `@code-help`, OpenAI Workspace Agents, Slack MCP Server 사례가 공통으로 보여주는 구조.
- 성격 다양성은 여러 LLM 공급자를 섞어서가 아니라 system prompt와 도구 접근 경계로 만든다. Claude API 하나로 수렴 가능하며 소규모 사용(월 1k 메시지 이하)이면 월 20~40 USD 수준.

## FAQ

### Q. Agent와 Skill은 파일·동작 수준에서 어떻게 다른가?

- Skill은 절차서·플레이북. 필요할 때만 로드되는 지시문이다.
- 호출 방식: `/이름`으로 직접 부르거나 Claude가 관련성을 판단해 자동 주입.
- Agent(Subagent)는 독립 context 윈도우 + 별도 system prompt + 도구 권한 + 메모리를 가진 역할자.
- Agent 정의에 여러 Skill을 묶어 처음부터 품고 시작할 수 있다.
- 선택 기준: 같은 절차를 누가 실행해도 결과가 같으면 Skill, 누적 컨텍스트·기억·도구 권한 분리가 필요하면 Agent.

### Q. 에이전트를 "정의"한다는 건 구체적으로 무엇을 정한다는 뜻인가?

- 역할: 이 에이전트가 잘해야 하는 일
- 도구: 어떤 검색기·파일·API에 접근하는가 (허용 목록)
- 기억: 무엇을 장기 기억으로 삼고, 무엇을 매 턴 검색할 것인가
- 출력 계약: 어떤 형식·길이·구조로 답해야 하는가
- 트리거: 어디서 호출되는가 — `/command`, `@mention`, 스케줄링, 다른 에이전트에서의 위임 등
- 이 다섯 가지가 실행 환경과 무관한 "에이전트 정의"의 본체다.

### Q. 같은 에이전트를 Claude Code와 슬랙에서 쓸 때 무엇이 같고 무엇이 다른가?

- 공통 재사용: Agent Core(목적·출력 계약·판단 기준)와 Tool/Memory Layer(MCP 서버로 추상화된 도구).
- 환경별로 새로 만드는 것: Runtime Adapter(이벤트 → Claude API 호출 래퍼)와 Interface(채널·DM·CLI).
- Claude Code: subagent 래퍼가 로컬 프로세스로 바로 실행.
- 슬랙: 외부 서버(Bolt 앱)가 이벤트를 받아 정의 파일을 system prompt로 변환 후 Claude API 호출, 결과를 슬랙 메시지로 렌더.

### Q. 슬랙에서 멀티에이전트는 실제로 어떤 용도로 쓰이는가?

- 채널에서 전문가 호출 — `@code-help`, `@literature-agent` 같은 역할자에게 질문하고 팀이 답을 검토.
- 티켓·이슈 자동 처리 — 답변 + 관련 문서 링크 + 새 이슈 생성까지 연결.
- 내부 지원 봇 — Notion·Google Drive·Salesforce 같은 소스에 붙여 FAQ 응답.
- "AI끼리 서로 논쟁하는 토론장" 패턴은 드물다. 주된 가치는 인간-에이전트 협업 인터페이스.

### Q. 슬랙 멀티에이전트를 실제로 구현할 때 핵심 테크닉은?

- 이벤트 수신: Slack Bolt 또는 `slack create agent` CLI 템플릿으로 앱 구성.
- 진입점: `@mention`, DM, assistant side panel 중 어느 것을 쓸지 정의.
- 정의 로딩: 이벤트를 받으면 `agents/<name>.md`를 읽어 system prompt와 허용 도구 목록으로 변환.
- 도구 추상화: MCP 서버로 검색기·메모리·외부 소스를 감싸두면 에이전트·환경 교체가 쉬워짐.
- 장기 기억: 긴 대화·누적 맥락은 제어 평면(로컬 markdown 등)에 저장, 슬랙은 스레드 식별자만 가진다.

### Q. Claude API 하나만으로 멀티에이전트를 운영할 수 있나? 비용 범위는?

- 가능. 성격 다양성은 모델 종류가 아니라 system prompt와 도구 접근 경계로 만든다.
- 여러 LLM API를 동시에 쓸 구조적 이유는 약함 — 같은 모델 위에서 역할만 다르게 주는 편이 관리가 쉽다.
- 소규모(월 1k 메시지 이하): Claude API 20~40 USD + 슬랙 무료 티어로 커버.
- 규모가 커지면 긴 context 비용과 MCP 도구 호출 수가 주된 비용 변수.
